## Context

Your design team built a prototype: a chat interface for your new AI feature. The CEO thinks it looks "very ChatGPT." The problem: 80% of your users don't want to chat. They want results.

As a PM, you need to understand that AI interfaces aren't just "normal UI plus chat." AI output is non-deterministic — and that changes everything: how you display results, how you build trust, and which interaction patterns work.

## Concept

### Non-Deterministic Interfaces

Traditional UX is deterministic: input X always produces output Y. A button always does the same thing. AI UX is probabilistic: input X produces a *distribution* of possible outputs.

This has direct consequences for your interface:
- Every AI output must be communicated as a **suggestion, not a fact**
- Interfaces must **surface variability**, not hide it
- Users need **regeneration options** ("Generate Again" buttons)
- **Confidence indicators** belong in the output, not in a debug view

### Chat vs. Structured Output

Not every AI interaction needs a chat window. The distinction matters:

| | Chat Interface | Structured Output |
|---|---|---|
| Strength | Exploratory, ambiguous, reasoning | Known inputs, validation, high volume |
| Weakness | Inefficient for repetitive tasks | Inflexible for open-ended questions |
| Example | "Analyze this contract" | Invoice classification with dropdowns |

**The rule:** "If it's a form, it should stay a form" (UX Planet). Forcing a form into a chat interface makes it worse, not better.

**Best practice:** Every form should be triggerable conversationally, and every agent should be able to hand off to a form. The pure chat paradigm is already outdated for complex workflows.

### Progressive Disclosure for AI

AI outputs contain a lot of information. Not all of it belongs on the first screen:

- **Level 1:** Core result — the answer the user needs
- **Level 2:** Confidence scores, sources, reasoning summary
- **Level 3:** Full chain of thought, raw data, alternatives

Microsoft's Copilot uses Dynamic Adaptive Behavior (DAB): the interface reshapes itself based on user behavior. Power users see more, casual users see less.

### Loading and Streaming UX

AI operations are inherently slower than CRUD. Streaming isn't optional:

- Users perceive streaming responses as **40-60% faster** than identical non-streaming responses
- Streaming signals "AI is working" — not just "loading"
- **Accessibility concern:** Streaming is problematic for screen readers. Plan for accessible alternatives.

## Framework

**AI UX pattern decision matrix:**

| Task | Pattern | Example Product |
|------|---------|----------------|
| Exploration, open-ended questions | Chat + Streaming | ChatGPT, Claude |
| Content creation | Chat + Side-by-side editor | ChatGPT Canvas, Claude Artifacts |
| Inline code assistance | Ghost text / Autocomplete | GitHub Copilot |
| Augmenting existing tools | Slash commands, inline AI | Notion AI (purple highlights) |
| Background intelligence | No visible AI UI | Linear (triage intelligence) |
| Visual generation | Grid + Variations | Midjourney (4-image grid) |
| Code generation + preview | Chat-to-output + live preview | v0 by Vercel |

**Design principles for choosing patterns:**

1. **Match modality to task** — not every AI needs chat
2. **Show AI is working** — streaming, skeleton screens, progress indicators
3. **Layer information** — progressive disclosure over information overload
4. **Accommodate variability** — "Generate Again," variants, sliders
5. **Respect existing workflows** — the Linear approach: AI enhances, instead of bolting on a chatbot

## Scenario

You're a PM at a project management tool (B2B SaaS, 8,000 teams). Your next feature: AI-powered sprint planning. The design team presents three options.

**The situation:**
- 65% of users use Board View, 25% List View, 10% Calendar
- Average sprint planning: 45 minutes per team per sprint
- User feedback: "We want faster planning, not another tool to learn"
- Competitor launched chat-based sprint planning, NPS at -12

**Options:**
1. **Chat interface:** Dedicated AI chat for sprint planning. "Plan my next sprint" and the AI suggests tickets
2. **Inline AI:** AI suggestions directly in the board. Purple-highlighted suggestions, one-click accept. No new interface
3. **Hybrid:** Chat for exploratory questions ("Which tickets are blocked?"), structured output for actual planning (drag-and-drop with AI ranking)

## Decide

<details>
<summary>How would you decide?</summary>

**The best decision:** Option 3 — Hybrid, with emphasis on inline.

**Why:**
- **User feedback is clear:** "Faster planning, not a new tool." That rules out a pure chat interface
- **Competitor signal:** Chat-only sprint planning has NPS -12. The market has already spoken
- **Board View dominates (65%):** AI must appear *in the board*, not beside it. Purple-highlighted suggestions with confidence score (Progressive Disclosure Level 1), reasoning on hover (Level 2)
- **Chat for the right moment:** "Show me all tickets that have been pushed for 3 sprints" is a valid chat query. But the answer should appear as a structured list, not prose
- **Expected impact:** 45 min down to ~25 min sprint planning via AI ranking + one-click accept. Chat-only would likely *increase* time (crafting prompts instead of clicking)

**Anti-patterns you're avoiding:**
- **Chat-for-everything:** Your competitor tried it. NPS -12
- **Hiding non-determinism:** AI suggestions must be recognizable as suggestions, with a "Shuffle" button for alternatives
- **Workflow disruption:** Users should stay in the board, not switch to a chat window
</details>

## Reflect

- **The interface must match the task, not the technology.** Chat is an interaction pattern, not a default. Familiar workflows deserve inline integration.
- **Non-determinism is a feature, not a bug.** Surface variability through regeneration, confidence scores, and alternatives — don't hide it behind a deterministic facade.
- **Progressive disclosure is mandatory for AI.** Level 1 for everyone, Level 2 on interest, Level 3 for power users. Microsoft's DAB approach points the direction.
- **Streaming is perception design.** 40-60% faster perceived at identical latency. But don't forget accessibility.
